Git

​ Git与Svn等工具差不多，都为版本控制工具，简单地说，版本控制是一种记录一个或若干文件内容变化，以便将来查阅特定版本修订情况的系统。Git 和其他版本控制系统的主要差别在于，Git 只关心文件数据的整体是否发生变化，而大多数其他系统则只关心文件内容的具体差异。

Git与Svn的区别

​ Git是分布式版本控制系统，像SVN这类就是集中式的版本控制系统。那什么是集中式，什么是分布式呢？

​ 先说集中式版本控制系统，他的版本库是集中存放在一台中央服务器的，而干活的时候，用的都是自己的电脑，所以要先从中央服务器取得最新的版本，然后开始干活，干完活了，再把自己的代码推送给中央服务器。

​ 中央服务器就好比是一个图书馆，你要改一本书，必须先从图书馆借出来，然后回到家自己改，改完了，再放回图书馆。集中式版本控制系统最大的问题就是必须联网才能工作。

​ 那分布式版本控制系统与集中式版本控制系统有何不同呢？只要提供一台电脑作为版本集中存的服务器放就够了，但这个服务器的作用仅仅是用来方便“交换”大家的修改，没有它也一样干活，只是交换修改不方便而已。而每一台电脑有各自独立的开发环境，每个人的电脑上都是一个完整的版本库，不需要联网，本地直接运行，相对集中式安全系数高很多。既然每个人电脑上都有一个完整的版本库，那多个人如何协作呢？比方说你在自己电脑上改了文件A，你的同事也在他的电脑上改了文件A，这时，你们俩之间只需把各自的修改推送给对方，就可以互相看到对方的修改了。

​ 可以这样说，如果你在一个不能连接网络的地方时，就像在飞机上，地下室，电梯里等，你仍然能够提交文件，查看历史版本记录，创建项目分支等，因为每个人的电脑上都是一个完整的版本库。

​ git是保存历史版本的完整文件，svn是保存文件差异。

Git工作区、版本库、暂存区

​ 工作区：电脑上能看到的文件夹，例如：D:\git\repository

​ 版本库：工作区中有一个隐藏的文件.git，就是git的版本库，版本库里的所有文件都被git所管理。

​ 暂存区：顾名思义，文件临时缓存的区域。

Git三种工作状态

​ modified：已修改，表示修改了文件，但还没保存到版本库中。

​ staged：已暂存，表示对一个已修改文件的当前版本做了标记，包含在下次提交的内容中。

​ committed：已提交，表示数据已安全的保存在本地仓库中。

Git安装

  1. 访问：http://www.git-scm.com/ ，下载相应版本，一直下一步直到安装完成。
  2. 安装成功后，在cmd里输入git –version，出现版本信息，则安装成功。
  3. 在任何地方右键点击git bash here进入git客户端

远程仓库

  1. 添加远程仓库，指令：git remote add [shortname] [url]

    ​ 注：把已有的本地仓库D:\git\repository与远程仓库关联，远程库的名字叫origin，这时git的默认叫法，可以改成其他的名字。

    1. 查看当前的远程仓库，指令：git remote

    ​ 注：当在第一步添加了一个远程仓库后，执行这个指令，会显示“origin”这个远程仓库的名字。

    1. 克隆远程仓库，指令：git clone [url]

    ​ 注：从零开发，最好先创建远程仓库，然后从远程仓库克隆，克隆成功后，在本地仓库会出现你克隆的内容文件。

    1. 在1添加完远程仓库，可以从远程仓库抓取数据，指令：git fetch [remote-name]
    1. 提交数据到远程仓库，指令：git push [remote-name] [branch-name]

    1. 查看远程仓库信息，指令：git remote show [remote-name]
    1. 重命名远程仓库，指令：git remote rename [old-remote-name] [new -remote-name]

Git分支

​ 分支在实际中有什么用呢？假设你准备开发一个新功能，但是需要两周才能完成，第一周你写了50%的代码，如果立刻提交，由于代码还没写完，不完整的代码库会导致别人不能干活了。如果等代码全部写完再一次提交，又存在丢失每天进度的巨大风险。

​ 现在有了分支，你就可以创建一个属于你自己的分支，别人看不到，还可以继续在原来的分支上正常工作，而你在自己的分支上干活，想提交就提交，直到开发完毕后，再一次性合并到原来的分支上，这样既安全，又不影响别人工作。

​ 其他版本控制系统如SVN等都有分支管理，但是用过之后你会发现，这些版本控制系统创建和切换分支比蜗牛还慢，简直让人无法忍受，结果分支功能成了摆设，大家都不去用。但Git的分支是与众不同的，无论创建、切换和删除分支，Git在1秒钟之内就能完成！无论你的仓库中是1个文件还是1万个文件。

分支使用流程

​ 在git里有一个分支叫做master，这个为master分支，指向最新更新的内容，还有一个指针head指向当前分支，即head指向master，master指向最新的提交。

​ 一开始master是一条线，git用master指向当前提交，用head指向master，就能确定当前分支，以及当前分支的提交点，每次提交，master都会向前移动一步，随着不断提交，master分支线也会越来越长。

​ 当我们创建了一个新的分支dev，指向master相同的提交，再把head指向dev，就表示当前分支在dev上，如果有新的提交上来，dev就往前一步，master不变，假如在dev上的工作完成了，就可以把dev合并到master上，直接把master指向dev的当前提交，就可以完成合并，再删掉dev分支就好了。我们的创建删除都是针对指针来操作，因此对工作内容不会进行改变。

分支使用

  1. 创建分支，指令：git branch [branch-name]

  2. 切换分支，指令：git checkout [branch-name]

  3. 查看分支，指令：git branch

    ​ 注：该指令会输出所有分支，并在当前分支前面打上*号。

  4. 在分支操作结束后，转回master，合并分支，指令：git merge [branch-name]

  5. 删除分支，指令：git branch -d [branch-name]

多人协作模式

  1. 从远程仓库克隆项目到本地；
  2. 在本地新增或修改代码并提交到本地仓库；
  3. 使用git push origin [branch-name]推送自己的修改；
  4. 如果3失败了，则因为远程分支比你的本地分支更新，需要先用git pull试图更新合并;
  5. 如果合并有冲突，则解决冲突，并在本地提交；
  6. 所有问题都解决后，再进行3的推送，就能成功。
